# CNews — Руководство по стилю

## Профиль издания

| Параметр | Значение |
|----------|----------|
| URL | cnews.ru |
| Аудитория | CIO, ИТ-директора, руководители ИТ-подразделений |
| Объём | Экспертные цитаты: 400-800 слов; Полные колонки: 1,500-2,500 слов |
| Технический уровень | 30-50% |
| Тон | Профессиональный, executive-level |
| Секции | «Взгляд эксперта», Аналитика, тематические разделы |

## Особенности CNews

### Формат издания
- Новостной портал с аналитикой
- Экспертные мнения интегрированы в новости
- Тематические разделы: Безопасность, Цифровизация, ИТ в банках
- Акцент на enterprise IT и B2G

### Целевая аудитория
- CIO крупных компаний
- Руководители ИТ-служб
- Государственные ИТ-директора
- Вендоры и интеграторы

## Форматы контента

### 1. Экспертная цитата (основной формат)

```
«[Имя], [Компания]: [Заголовок-тезис]»

[2-4 абзаца экспертного мнения]
```

**Объём:** 400-800 слов (2,500-5,000 символов)

### 2. Полная авторская колонка

**Объём:** 1,500-2,500 слов
**Структура:** Как в деловых СМИ, но с большим техническим фокусом

## Структура экспертной цитаты

```
ЗАГОЛОВОК
├── Формат: «[Имя], [Компания]: [Тезис]»
├── Тезис должен быть конкретным
└── Пример: «Игорь Петров, VK Cloud: Облачная безопасность требует нового подхода»

ОСНОВНОЙ ТЕЗИС (1 абзац)
├── Главная мысль эксперта
├── Почему это важно
└── Связь с текущей повесткой

АРГУМЕНТАЦИЯ (1-2 абзаца)
├── Данные и факты
├── Примеры из практики
└── Технический контекст

ВЫВОД / РЕКОМЕНДАЦИЯ (1 абзац)
├── Что делать ИТ-директорам
└── Взгляд в будущее
```

## Стилистические требования

### Язык
- Деловой, но не сухой
- Экспертная позиция, не нейтральная
- Допустимы оценочные суждения
- Технические термины без избыточных объяснений

### Тональность
- Авторитетная
- Отраслевая экспертиза
- Практичная (не теоретическая)
- Для принятия решений

## Формат представления автора

```
[Полное имя], [должность] [компания]
```

**Примеры:**
- Игорь Петров, директор по безопасности VK Cloud
- Мария Сидорова, руководитель направления цифровизации VK Cloud

## Примеры статей

### Пример 1: Экспертная цитата (безопасность)

**Заголовок:** Олег Бойко, VK Cloud: Многоуровневая защита — фундамент облачной безопасности

**Текст:**

Компании всё чаще доверяют облачным провайдерам критичные системы и персональные данные. При этом вопрос «насколько безопасно облако» остаётся одним из главных барьеров для миграции. Ответ: облако безопасно ровно настолько, насколько глубоко продуман подход к защите.

В VK Cloud мы применяем принцип Defense in Depth — многоуровневую защиту, где каждый уровень независим. Физическая безопасность ЦОД (контроль доступа, видеонаблюдение, Tier 3), сетевая защита от DDoS-атак, безопасность операционной системы (журналирование, аутентификация), защита на уровне приложений.

Наша платформа обрабатывает 2,5–3,5 млн событий безопасности в секунду. В октябре 2024 года VK Cloud получила сертификат ФСТЭК России, что позволяет клиентам строить инфраструктуру для работы с персональными данными по УЗ-1.

Но главные угрозы — не хакеры извне. По статистике инцидентов, фишинг и ошибки конфигурирования остаются основными векторами атак. Облачная платформа обеспечивает технологический базис, но ответственность за уровень приложений лежит на клиенте.

**Автор:** Олег Бойко, директор по информационной безопасности VK Cloud

**Оценка:** Соответствует формату CNews. Конкретный тезис, метрики, compliance-контекст, практическая рекомендация.

### Пример 2: Экспертная цитата (цифровизация)

**Заголовок:** Дмитрий Лазаренко, VK Cloud: Managed-сервисы становятся стандартом для enterprise

**Текст:**

Структура потребления облачных услуг меняется. Если раньше основным драйвером роста был IaaS — аренда виртуальных машин и хранилищ, то сегодня быстрее всего растёт сегмент PaaS и managed-сервисов.

По данным VK Cloud, выручка направления Managed Services выросла на 82% год к году. Клиенты предпочитают managed-базы данных, Kubernetes, Kafka — не потому что не умеют настраивать сами, а потому что стоимость содержания команды DBA превышает стоимость сервиса в 2-3 раза.

Особенно заметен этот сдвиг в финансовом секторе. Требования ЦБ к отказоустойчивости и резервному копированию проще выполнить, когда за инфраструктуру отвечает провайдер с сертификатами ГОСТ Р 57580. Это снимает с ИТ-службы значительную часть compliance-нагрузки.

Managed Kubernetes востребован каждым пятым клиентом VK Cloud. В 2025 году ожидаем, что managed-сервисы станут стандартом де-факто. Вопрос сместится от «делать самим или отдать провайдеру» к «какие workloads оставить у себя, а какие вынести».

**Автор:** Дмитрий Лазаренко, директор по продукту VK Cloud

**Оценка:** Хороший пример для CNews. Отраслевой фокус, конкретные метрики, регуляторный контекст, прогноз.

### Пример 3: Полная авторская колонка

**Заголовок:** Как выбрать облачного провайдера: практический чеклист для ИТ-директора

**Текст:**

Выбор облачного провайдера — решение, которое будет влиять на компанию годами. Миграция между провайдерами — процесс дорогой и болезненный. При этом большинство тендеров на облачные услуги проводятся по критериям, которые не отражают реальных потребностей бизнеса.

За последние пять лет я участвовал в десятках проектов по выбору облачной инфраструктуры — и как вендор, и как консультант. На основе этого опыта предлагаю практический подход к оценке провайдеров.

## Критерии первого уровня: must have

Эти параметры должны быть выполнены безусловно. Если провайдер не соответствует хотя бы одному — он не должен рассматриваться.

**1. Локализация данных**

Для большинства российских компаний данные должны храниться на территории РФ. Это требование 152-ФЗ для персональных данных и здравый смысл для всех остальных.

Что проверять:
- Физическое расположение дата-центров
- Возможность выбора региона размещения
- Гарантии отсутствия репликации за пределы РФ

**2. SLA и компенсации**

SLA — не маркетинговая цифра, а юридическое обязательство. Важно не только значение (99.9% vs 99.95%), но и механизм компенсации.

Что проверять:
- Что именно покрывает SLA (compute, storage, network отдельно)
- Размер компенсации за нарушение
- Процедура заявки на компенсацию
- История реальных инцидентов (можно запросить)

**3. Сертификации и compliance**

Для регулируемых отраслей (финансы, медицина, госсектор) сертификации обязательны.

Что проверять:
- Аттестация по 152-ФЗ
- ГОСТ Р 57580 для финсектора
- Сертификаты ISO 27001, SOC 2
- Лицензии ФСТЭК и ФСБ (если требуются)

## Критерии второго уровня: важные

**4. Технический стек**

Совместимость с вашими текущими инструментами снижает стоимость миграции.

Что проверять:
- Поддержка Terraform, Ansible
- Kubernetes-совместимость (CNCF certified)
- API-совместимость с S3, OpenStack
- Интеграция с CI/CD системами

**5. Поддержка и экспертиза**

Техническая поддержка — критичный фактор, который часто недооценивают.

Что проверять:
- Время реакции по уровням критичности
- Наличие выделенного менеджера
- Экспертиза в вашей отрасли
- Референсы похожих проектов

**6. Ценообразование и прозрачность**

Облачные счета печально известны непредсказуемостью.

Что проверять:
- Калькулятор стоимости (насколько точен)
- Скрытые платежи (трафик, API-запросы)
- Committed use discounts
- Возможность бюджетирования

## Критерии третьего уровня: конкурентные преимущества

**7. Roadmap и развитие**

Провайдер должен развиваться вместе с вашими потребностями.

**8. Экосистема партнёров**

Интеграторы, ISV, консультанты — кто поможет с внедрением?

**9. Финансовая устойчивость**

Провайдер должен существовать дольше, чем ваш проект.

## Процесс выбора: рекомендуемый подход

1. **Shortlist** (2 недели): Отсеять по критериям первого уровня
2. **RFI** (2-3 недели): Собрать детальную информацию по второму уровню
3. **POC** (4-6 недель): Пилотный проект с 2-3 финалистами
4. **Финальная оценка** (2 недели): TCO, референсы, переговоры
5. **Решение**

## Чеклист для тендерной документации

□ Требования к локализации данных
□ Минимальный SLA и механизм компенсаций
□ Список необходимых сертификаций
□ Требования к совместимости (API, IaC)
□ Требования к поддержке (SLO, языки, каналы)
□ Формат ценообразования (фиксированный / usage-based)
□ Требования к миграции (инструменты, поддержка)
□ Условия выхода (data portability, lock-in)

## Заключение

Выбор облачного провайдера — стратегическое решение. Потраченные на тщательную оценку 2-3 месяца окупятся многократно: меньше рисков, меньше неожиданностей, меньше необходимости мигрировать через год.

Главный совет: не выбирайте по цене. Самый дешёвый провайдер часто оказывается самым дорогим, когда учитываешь скрытые расходы на поддержку, миграцию и простои.

**Автор:** Дмитрий Соколов, коммерческий директор VK Cloud. 12 лет опыта в корпоративных продажах ИТ-решений.

**Оценка:** Хороший пример полной колонки для CNews. Практическая ценность для ИТ-директоров, структурированный чеклист, экспертный тон.

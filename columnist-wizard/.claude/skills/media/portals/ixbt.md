# iXBT.com — Руководство по стилю

## Профиль издания

| Параметр | Значение |
|----------|----------|
| URL | ixbt.com, ixbt.com/live |
| Аудитория | IT-специалисты, сисадмины, техноэнтузиасты |
| Объём | 1,500-3,000+ слов |
| Технический уровень | 70-90% |
| Тон | Экспертный, детальный, честный |
| Трафик | 50M+ просмотров/месяц |
| Контакт редакции | editors@ixbt.com |

## Официальные требования редакции

**Источники:** [ixbt.com/authors.html](https://www.ixbt.com/authors.html), [ixbt.com/live/page/about](https://www.ixbt.com/live/page/about/)

### Формат подачи материала
- Формат файла: RTF или Microsoft Word
- Изображения: JPG, GIF, PNG
- Архивы: WinZIP или WinRAR
- URL в тексте должны быть кликабельными ссылками
- Срок рассмотрения: до 14 дней

### Требования к контенту
```
✓ Только оригинальные материалы
✓ Объективный и аргументированный тон
✓ Правильные ссылки и атрибуция источников
✓ Самодостаточный текст (без внешних ссылок, заменяющих статью)
```

### Категорически запрещено
```
❌ Плагиат или ранее опубликованные материалы
❌ Перепечатка без указания источника
❌ Оскорбительная лексика
❌ Реклама, пресс-релизы, промо-тексты (БАН автора и компании)
❌ Множественные аккаунты
❌ Недостоверные или непроверенные факты
❌ Нарушение авторских прав на изображения
```

### Требования к иллюстрациям
- **Только собственные** изображения или с правами на использование
- Чужие иллюстрации без лицензии запрещены
- Ужесточённые требования из-за авторских прав

### Система статусов на iXBT Live
```
Начинающий автор → Автор → Постоянный автор

- Начинающий: все посты на премодерации
- Автор: 2+ поста в тематических блогах
- Постоянный: выдаётся модераторами за качество и соблюдение правил
```

### Правило подборок
- Не более 4 подборок подряд
- Минимум 1 из 5 постов — обзор или статья
- Не более 2 подборок в неделю

## Стилистические требования

### Язык
- Технический, но понятный
- Допустим профессиональный жаргон
- Точность формулировок критична
- Оценочные суждения допустимы

### Что ценится
- Оригинальная методология тестирования
- Глубина анализа
- Честность об ограничениях
- Практическая применимость
- Уникальные данные

### Чего избегать
- Маркетинговая риторика
- Поверхностное изложение
- Необоснованные выводы
- «Капитанские» истины

## Форматирование

### Обязательные элементы
- Качественные изображения (фото, скриншоты)
- Графики и диаграммы для данных
- Таблицы для сравнений
- Подписи к изображениям

### Типографика
- Тире: — (не -)
- Кавычки: «» или ""
- Единицы измерения через неразрывный пробел

## Примеры статей

### Пример 1: Техническое исследование

**Заголовок:** Реальная производительность NVMe SSD в облаке: тестируем 5 российских провайдеров

**Текст:**

Маркетинговые материалы облачных провайдеров обещают «NVMe-скорости» для блочного хранилища. Но что стоит за этими обещаниями? Мы провели независимое тестирование дисковой подсистемы у пяти крупнейших российских облачных провайдеров.

## Методология

Для каждого провайдера мы создали идентичные виртуальные машины:
- 4 vCPU
- 16 GB RAM
- 100 GB SSD (максимально доступный «быстрый» тип)
- Ubuntu 22.04 LTS

Тестирование проводилось с помощью fio — стандартного инструмента для измерения производительности дисков.

```bash
# Тест случайного чтения 4K
fio --name=random-read \
    --ioengine=libaio \
    --iodepth=32 \
    --rw=randread \
    --bs=4k \
    --direct=1 \
    --size=10G \
    --numjobs=4 \
    --runtime=60 \
    --group_reporting

# Тест случайной записи 4K
fio --name=random-write \
    --ioengine=libaio \
    --iodepth=32 \
    --rw=randwrite \
    --bs=4k \
    --direct=1 \
    --size=10G \
    --numjobs=4 \
    --runtime=60 \
    --group_reporting

# Тест последовательного чтения
fio --name=seq-read \
    --ioengine=libaio \
    --iodepth=32 \
    --rw=read \
    --bs=1M \
    --direct=1 \
    --size=10G \
    --numjobs=1 \
    --runtime=60 \
    --group_reporting
```

Каждый тест проводился 5 раз в разное время суток, результаты усреднялись.

## Анализ результатов

Большинство провайдеров показывают результаты, близкие к заявленным. Исключение — Провайдер C, где реальная производительность значительно ниже маркетинговых обещаний.

**Важное наблюдение:** У всех провайдеров производительность падала в пиковые часы (10:00-12:00 и 15:00-17:00 по Москве) на 5-15%. Это ожидаемо для shared-инфраструктуры.

### Стабильность

Помимо средних значений, важна стабильность. Измеряли p99 latency:

| Провайдер | p99 Read Latency | p99 Write Latency |
|-----------|------------------|-------------------|
| Провайдер A | 2.1 ms | 3.8 ms |
| Провайдер B | 1.8 ms | 3.2 ms |
| Провайдер C | 12.4 ms | 28.6 ms |
| Провайдер D | 1.9 ms | 3.5 ms |
| Провайдер E | 1.7 ms | 3.1 ms |

У Провайдера C наблюдаются значительные «хвосты» latency, что критично для баз данных и приложений, чувствительных к задержкам.

## Ограничения исследования

1. Тестировались только стандартные SSD-диски, не high-performance tier
2. Тестирование проводилось в одном регионе
3. Не тестировались сценарии под нагрузкой
4. Результаты могут отличаться для других конфигураций VM

## Выводы

1. **Маркетинг vs реальность:** Большинство провайдеров честны в своих заявлениях (отклонение <10%). Но есть исключения.

2. **p99 важнее среднего:** Для production-систем смотрите на хвостовые latency, не на средние значения.

3. **Время суток имеет значение:** Планируйте нагрузочное тестирование на пиковые часы.

4. **Проводите свои тесты:** Наша методология — отправная точка. Тестируйте под ваши сценарии.

### Пример 2: Мнение / Opinion

**Заголовок:** Почему я перестал рекомендовать Docker Desktop для разработки

**Текст:**

Три года я убеждал всех вокруг, что Docker Desktop — лучший способ работать с контейнерами на macOS и Windows. Сегодня я изменил мнение. Вот почему.

## Контекст

Моя работа — помогать командам настраивать dev-окружения. За последний год я видел десятки проблем, связанных с Docker Desktop. И всё чаще рекомендую альтернативы.

## Проблема #1: Производительность на macOS

Docker Desktop использует виртуальную машину Linux под капотом. На macOS (особенно на Intel) это создаёт overhead, который заметен на реальных проектах.

Типичный сценарий:
- Node.js проект с hot reload
- 50,000 файлов в node_modules
- Время пересборки при изменении файла: 15-30 секунд

С нативным Linux (WSL2 на Windows, Colima на macOS): 2-5 секунд.

**Измерения:** webpack build на типичном React-проекте:
- Docker Desktop: 45 секунд
- Colima + Docker: 18 секунд
- Нативный Linux: 12 секунд

## Проблема #2: Лицензирование

С 2021 года Docker Desktop требует платной подписки для компаний более 250 сотрудников или с выручкой более $10M.

Для enterprise это означает:
- $5-21/месяц за пользователя
- Процесс согласования закупки
- Аудит использования

При этом сам Docker Engine остаётся бесплатным. Платите вы за GUI и интеграцию.

## Проблема #3: Ресурсоёмкость

Docker Desktop по умолчанию резервирует:
- 2 GB RAM (часто увеличивают до 4-8 GB)
- CPU cores для VM

На 16 GB ноутбуке это заметно. Особенно когда открыто 15 вкладок Chrome и IDE.

## Альтернативы, которые я рекомендую

### Для macOS: Colima

```bash
brew install colima docker
colima start --memory 4 --cpu 2
docker context use colima
```

Плюсы:
- Бесплатный
- Быстрее Docker Desktop
- Меньше ресурсов
- Lima под капотом (хорошо оптимизирован)

Минусы:
- Нет GUI (для кого-то это плюс)
- Требует настройки

### Для Windows: WSL2 + Docker Engine

```powershell
wsl --install -d Ubuntu
# Внутри Ubuntu:
curl -fsSL https://get.docker.com | sh
```

Плюсы:
- Нативная производительность Linux
- Бесплатный
- Интеграция с Windows filesystem

Минусы:
- Нужен WSL2
- Настройка networking может быть сложной

### Для production-like окружений: Podman

```bash
brew install podman
podman machine init
podman machine start
```

Плюсы:
- Совместим с Docker CLI
- Rootless по умолчанию
- Бесплатный
- Нет daemon

Минусы:
- Compose поддержка хуже
- Некоторые edge cases работают иначе

## Когда Docker Desktop всё ещё имеет смысл

- Для новичков (GUI помогает)
- Когда компания уже платит за лицензию
- Для Kubernetes-разработки (встроенный кластер)
- Когда время настройки критичнее производительности

## Итого

Docker Desktop — неплохой продукт. Но не единственный вариант и не лучший для всех сценариев.

Если вы:
- Работаете на macOS и страдаете от тормозов — попробуйте Colima
- Работаете на Windows — WSL2 + Docker Engine быстрее и бесплатно
- Хотите rootless security — смотрите на Podman

Главный вывод: инструменты нужно выбирать под задачу, а не по инерции.

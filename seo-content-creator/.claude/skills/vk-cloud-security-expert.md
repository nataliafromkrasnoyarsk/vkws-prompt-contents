## WRITING GUIDELINES

### Терминология

**Используй правильную терминологию:**

| Правильно | Неправильно |
|-----------|-------------|
| информационная безопасность | инфобез, кибербез (в тексте) |
| уязвимость | дыра, баг безопасности |
| злоумышленник, атакующий | хакер (нейтральный контекст) |
| инцидент безопасности | взлом (если не подтверждён) |
| аутентификация | авторизация (когда речь о проверке личности) |
| авторизация | аутентификация (когда речь о правах доступа) |
| шифрование | кодирование (разные вещи) |
| персональные данные | личные данные (в контексте 152-ФЗ) |

### Стиль написания

```yaml
tone:
  - Профессиональный, но доступный
  - Без запугивания читателя
  - С конкретными рекомендациями
  - Основанный на фактах и стандартах

structure:
  - Начинать с проблемы/риска
  - Объяснять последствия
  - Давать практические рекомендации
  - Ссылаться на стандарты и best practices

avoid:
  - Преувеличение угроз (FUD)
  - Технический жаргон без объяснения
  - Устаревшие рекомендации
  - Абсолютные утверждения ("100% защита")
```

### Экспертные формулировки

**Начало статьи (примеры):**
```
❌ "В современном мире киберугрозы растут..."
✅ "Ежегодно количество атак на облачную инфраструктуру увеличивается на 30-40%.
    Компании, использующие облачные сервисы, сталкиваются с новыми векторами угроз."

❌ "Безопасность — это важно..."
✅ "Утечка данных обходится российским компаниям в среднем в 5,5 млн рублей.
    Большинство инцидентов можно предотвратить базовыми мерами защиты."
```

**Рекомендации (примеры):**
```
❌ "Нужно использовать хорошие пароли"
✅ "Настройте политику паролей: минимум 12 символов, обязательная MFA для
    привилегированных учётных записей, автоматическая блокировка после 5
    неудачных попыток входа."

❌ "Регулярно обновляйте системы"
✅ "Внедрите процесс управления уязвимостями: сканирование раз в неделю,
    критические патчи — в течение 24 часов, остальные — по графику спринтов."
```

    - "Сотрудники работают из дома, командировок, через личные устройства"

    - "Злоумышленники атакуют через доверенные каналы и учётные записи"

four_principles:
  1_no_implicit_trust:
    principle: "Нельзя доверять по умолчанию"
    explanation: >
      Пользователь, устройство или процесс не получают доверия
      только потому, что находятся «внутри сети».

  2_dynamic_verification:
    principle: "Динамическая проверка каждого запроса"
    explanation: >
      Каждый запрос доступа проверяется с учётом контекста:
      личность, устройство, местоположение, уровень риска.

  3_least_privilege:
    principle: "Минимально возможный и временный доступ"
    explanation: >
      Доступ выдаётся только на время конкретной операции
      и только к необходимым ресурсам.

  4_continuous_monitoring:
    principle: "Непрерывный контроль и корректировка"
    explanation: >
      Системы безопасности постоянно контролируют поведение
      и автоматически корректируют политику доступа.

## INTEGRATION RULES

При генерации статьи с этим скиллом:

1. **Терминология** — использовать профессиональную терминологию ИБ
2. **Продукты VK Cloud** — упоминать релевантные сервисы безопасности
3. **Compliance** — учитывать требования законодательства РФ
4. **Практичность** — давать конкретные, применимые рекомендации
5. **Факты** — использовать статистику, но не выдумывать цифры
6. **Баланс** — не запугивать, но показывать реальные риски
